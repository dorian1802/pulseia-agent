## Введение: Реальность ИИ-агентов в продакшене

Переход от прототипа к продакшену — момент, когда большинство проектов ИИ-агентов сталкиваются с серьёзными проблемами. В разработке агент работает на тщательно отобранных тестовых случаях. В продакшене он сталкивается с непредсказуемыми вводами, крайними случаями, вариациями контекста и пользователями, систематически исследующими пределы системы.

Статистика красноречива: согласно исследованию Gartner, опубликованному в 2025 году, примерно **70% проектов автономных ИИ-агентов** сталкиваются со значительными проблемами в первые три месяца работы в продакшене. Причины разнообразны, но имеют общее: их часто **трудно обнаружить** без адекватной наблюдаемости.

Эта статья анализирует наиболее частые режимы сбоев и показывает, как [Langfuse](/blog/langfuse-surveiller-agents-ia) позволяет их обнаруживать, диагностировать и быстро исправлять.

## 7 наиболее частых режимов сбоев

### 1. Тихая деградация качества (Silent Quality Drift)

Вероятно, самый коварный режим сбоя. Агент продолжает отвечать без технических ошибок, но качество ответов постепенно ухудшается.

**Типичные причины:**
- Тихое обновление базовой модели провайдером
- Эволюция базы знаний (новые документы, устаревшие документы)
- Изменение в паттернах запросов пользователей (более сложные вопросы)
- Дрифт температуры в векторных эмбеддингах

**Как обнаружить с Langfuse:**

```python
# Настроить автоматические оценки качества
from langfuse import Langfuse

langfuse = Langfuse()

# На каждую трассу в продакшене рассчитывать оценку качества
trace = langfuse.trace(name="agent-support")

# Автоматическая оценка LLM-as-Judge
trace.score(name="quality", value=quality_score)
trace.score(name="faithfulness", value=faithfulness_score)

# Настроить оповещение по скользящему среднему
# Если average(quality, last_100_traces) < 3.5, отправить оповещение
```

Дашборд Langfuse позволяет визуализировать тренд оценок качества на скользящих окнах, делая видимым ухудшение, которое иначе осталось бы незаметным.

### 2. Взрывные рост затрат (Cost Spikes)

ИИ-агенты, использующие мощные модели вроде GPT-4o или Claude Opus, могут столкнуться с внезапным ростом затрат.

**Типичные причины:**
- Бесконечный цикл рассуждений (агент крутится без сходимости)
- Аномально длинные запросы, потребляющие тысячи токенов
- Пользователь или бот, отправляющий тысячи запросов
- Изменение маршрутизации, направляющее простые задачи на дорогую модель

**Как обнаружить с Langfuse:**

```python
# Оповещение об аномально длинных трассах
trace = langfuse.trace(
    name="agent-rag",
    metadata={"max_tokens_budget": 5000}
)

# Если трасса потребляет более 5000 токенов, это ненормально
generation = trace.generation(
    name="gpt4-call",
    model="gpt-4o",
    usage={"input": 8000, "output": 2000, "total": 10000}  # Аномально!
)

# Настроить оповещение в дашборде:
# Cost per trace > 0.10 $ -> Оповещение Slack
# Daily cost > 50 $ -> Оповещение PagerDuty
```

Для полного мониторинга затрат обратитесь к нашему руководству по [мониторингу ИИ-агентов с Langfuse](/blog/langfuse-surveiller-agents-ia).

### 3. Бесконечные циклы и таймауты

Автономные агенты особенно уязвимы к бесконечным циклам: агент вызывает инструмент, получает неудовлетворительный результат, снова вызывает тот же инструмент с аналогичными параметрами и так далее.

**Типичные причины:**
- Плохо спроектированный промпт без чётких условий выхода
- Дефектный инструмент, возвращающий перемежающиеся ошибки
- Переполнение контекстного окна: агент теряет след того, что уже пробовал

**Как обнаружить с Langfuse:**

```
Трасса: "Агент отладки кода"
├── Span: "Вызов инструмента search_code" (1.2s) ✅
├── Span: "Вызов инструмента search_code" (1.1s) ⚠️ Дубликат
├── Span: "Вызов инструмента search_code" (1.3s) ⚠️ Дубликат
├── Span: "Вызов инструмента search_code" (1.0s) ⚠️ Дубликат
├── Span: "Вызов инструмента search_code" (1.2s) 🔴 Цикл!
└── TIMEOUT через 60 секунд
```

Иерархическая структура трасс Langfuse делает циклы немедленно видимыми: если вы видите повторение одного и того же span'а, вы знаете, что есть проблема.

### 4. Каскадные галлюцинации

В мультиагентных системах галлюцинация одного агента может распространиться на последующих агентов, принимающих эту информацию за достоверную.

**Типичные причины:**
- Первый агент производит некорректную информацию
- Последующие агенты доверяют результату без проверки
- Отсутствие механизма перекрёстной валидации

**Конкретный сценарий:**

```
Агент 1: "Гарантия этого продукта — 5 лет"  ← Галлюцинация
  ↓
Агент 2: "Согласно нашим данным, гарантия — 5 лет,
           вот условия..."  ← Распространение
  ↓
Пользователь: Получает ложную информацию
```

**Как обнаружить с Langfuse:**

```python
# Оценка доверия для каждого агента
agent1_trace = langfuse.trace(name="agent-recherche")
agent1_trace.score(name="confidence", value=0.45)  # Низкий балл

# Если confidence < 0.6, предупредить следующий агент
# или переключиться на механизм верификации
```

Для систематической оценки достоверности ответов см. наш сравнительный анализ [Ragas vs Promptfoo](/blog/ragas-vs-promptfoo).

### 5. Сбои рассуждений (Reasoning Failures)

Агенты на основе моделей рассуждений (Chain-of-Thought, ReAct) могут отказывать тонким образом: рассуждение логически некорректно, но ответ кажется правдоподобным.

**Типичные причины:**
- Путаница между похожими концепциями
- Необоснованные логические скачки в рассуждении
- Предвзятость подтверждения (агент игнорирует противоречащие свидетельства)
- Излишняя уверенность в результатах дефектных инструментов

**Как обнаружить:**

Изучая трассы рассуждений в Langfuse, можно выявить точки, где рассуждение даёт сбой. Ключ — **трассировать каждый шаг цепи рассуждений** как отдельный span.

### 6. Ошибки интеграции с инструментами

Агенты, использующие внешние инструменты (API, базы данных, веб-сервисы), уязвимы к сбоям этих сервисов.

**Типичные причины:**
- Сторонний API не работает или медленный
- Изменение формата ответа API
- Проблемы аутентификации (истёкшие токены)
- Превышение rate limiting

**Как обнаружить с Langfuse:**

```python
# Трассировать каждый вызов инструмента с результатом
tool_span = trace.span(
    name="tool-call-weather-api",
    input={"city": "Париж", "date": "2026-04-19"},
    output={"error": "503 Service Unavailable"},
    status="error"
)

# Ошибки инструментов отображаются красным в дашборде
# Настроить оповещение: tool_error_rate > 10 % -> Оповещение
```

### 7. Переполнение контекстного окна

Когда история разговора или извлечённые документы превышают размер контекстного окна, модель молча усекает информацию, что может привести к неполным или несогласованным ответам.

**Типичные причины:**
- Длинные разговоры без механизма резюмирования
- Слишком объёмный retrieval (слишком много возвращённых документов)
- Ошибка в расчёте размера токенов

**Как обнаружить с Langfuse:**

```python
# Отслеживать коэффициент использования контекстного окна
generation = trace.generation(
    name="gpt4-response",
    model="gpt-4o",
    usage={"input": 124000, "output": 500, "total": 124500}
)

# Если input_tokens приближается к 128000 (лимит GPT-4o), предупредить
# Индикатор: context_window_usage = input / max_tokens
# Если > 0.9, риск усечения
```

## Стратегии отладки с Langfuse

### Стратегия 1: Пост-мортем на основе трасс

Когда пользователь сообщает о проблеме, первый шаг — найти соответствующую трассу в Langfuse.

```python
# Поиск трассы по пользователю и дате
traces = langfuse.get_traces(
    user_id="client-signale-incident",
    from_timestamp="2026-04-18T00:00:00Z",
    to_timestamp="2026-04-19T00:00:00Z"
)

# Анализ каждого span проблемной трассы
for trace in traces:
    for span in trace.spans:
        print(f"Span: {span.name}")
        print(f"Длительность: {span.duration_ms}ms")
        print(f"Статус: {span.status}")
        if span.status == "error":
            print(f"Ошибка: {span.error_message}")
```

Этот подход позволяет реконструировать точный путь запроса и идентифицировать точку сбоя.

### Стратегия 2: Агрегированный анализ сбоев

Вместо анализа трасс по одной, Langfuse позволяет выявлять паттерны сбоев в масштабе.

- **Топ ошибок** — Какие ошибки наиболее частые?
- **Распределение задержек** — Есть ли пик задержки в определённые часы?
- **Корреляции** — Коррелируют ли ошибки инструментов с галлюцинациями?

### Стратегия 3: Тестирование в продакшене (Shadow Mode)

Для крупных изменений (новый промпт, новая модель, новый пайплайн) тестирование в shadow mode — самая безопасная стратегия:

1. Развёрнуть новую версию параллельно со старой
2. Отправлять одни и те же запросы обеим версиям
3. Сравнивать результаты и метрики в Langfuse
4. Переключаться только когда новая версия подтверждена как лучшая

```python
# Shadow mode: две трассы для каждого запроса
prod_trace = langfuse.trace(
    name="agent-rag-v1",
    version="1.0.0"
)

shadow_trace = langfuse.trace(
    name="agent-rag-v2",
    version="2.0.0",
    metadata={"shadow_mode": True}
)

# Сравнить оценки в дашборде
```

## Полная конфигурация Langfuse для мониторинга

### Шаг 1: Инструментирование агента

```python
from langfuse.decorators import observe
from langfuse import Langfuse

langfuse = Langfuse()

@observe(name="agent-production")
def agent_pipeline(user_query: str, user_id: str):
    """Полный агент с инструментированием Langfuse."""

    # Извлечение
    docs = retrieve_documents(user_query)
    # Декоратор @observe автоматически захватывает этот span

    # Генерация
    response = generate_response(user_query, docs)

    # Автоматическая оценка
    quality = evaluate_quality(user_query, response, docs)

    # Оценка качества, внедрённая в трассу
    langfuse.score_current_trace(
        name="auto_quality",
        value=quality
    )

    return response
```

### Шаг 2: Настройка оповещений

| Оповещение | Условие | Канал | Приоритет |
|---|---|---|---|
| Низкое качество | avg(quality, 50 трасс) < 3.0 | Slack | Высокая |
| Высокие затраты | daily_cost > 100 $ | Slack + Email | Высокая |
| Задержка P95 | latency_p95 > 10s | Slack | Средняя |
| Уровень ошибок | error_rate > 5 % | PagerDuty | Критическая |
| Обнаружен цикл | spans_same_name > 5 | Slack | Высокая |
| Насыщение контекста | context_usage > 0.9 | Slack | Средняя |

### Шаг 3: Дашборды мониторинга

Настройте как минимум три дашборда:

1. **Операционный дашборд** — Задержка, уровень ошибок, объём запросов, доступность
2. **Дашборд качества** — Оценки качества, уровень галлюцинаций, релевантность
3. **Финансовый дашборд** — Затраты по модели, по агенту, по пользователю, тренды

## Примеры реальных сбоев и их исправления

### Случай 1: Агент поддержки, галлюцинировавший после обновления

**Симптом**: Агент клиентской поддержки внезапно начал выдумывать политики возврата после тихого обновления GPT-4o.

**Обнаружение через Langfuse**: Оценка faithfulness упала с 0.92 до 0.61 за две недели, что видно на дашборде качества.

**Исправление**: Системный промпт был усилен явными инструкциями опираться только на предоставленные документы. Добавлен регрессионный тест [Promptfoo](/blog/ragas-vs-promptfoo) для вопросов о политике.

### Случай 2: Автономный агент в бесконечном цикле

**Симптом**: Агент отладки кода крутился в цикле 60 секунд до таймаута, потребляя тысячи токенов.

**Обнаружение через Langfuse**: Трасса показала 12 последовательных вызовов одного и того же инструмента поиска кода с почти идентичными параметрами.

**Исправление**: Добавлен механизм обнаружения циклов в промпт агента и жёсткий лимит в 5 последовательных вызовов одного и того же инструмента.

### Случай 3: Взрыв затрат от одного пользователя

**Симптом**: Ежедневные затраты утроились за один день.

**Обнаружение через Langfuse**: Финансовый дашборд показал, что один пользователь (скрапинг-бот) был ответственен за 85% затрат.

**Исправление**: Внедрён rate limiting по пользователю и оповещение о стоимости по пользователю (макс. 5 $/день).

## Обновление — Апрель 2026

Практики мониторинга ИИ-агентов значительно созрели:

- **Самоисцелящиеся агенты** — Некоторые фреймворки теперь позволяют агентам автоматически корректироваться, когда Langfuse обнаруживает сбой (переключение на резервную модель, упрощение промпта, сокращение контекста)
- **Автоматическая корреляция** — Langfuse v3 предлагает автоматическую корреляцию между метриками качества и изменениями конфигурации, позволяя выявлять первопричину без ручного расследования
- **Непрерывный бенчмарк** — Метрики [Ragas](/blog/ragas-vs-promptfoo) могут выполняться непрерывно на выборке продакшен-трасс, замыкая цикл между мониторингом и оценкой
- **Межагентная наблюдаемость** — Для мультиагентных систем (как [AutoGPT](/blog/autogpt-bilan-2026) или [OpenClaw](/blog/openclaw-test-exclusif)) Langfuse теперь предлагает распределённую трассировку, отслеживающую запросы через несколько агентов
- **Предиктивные оповещения** — На основе ML-моделей, изучающих нормальные паттерны вашего агента и предупреждающих до достижения порога

Область быстро развивается, но фундаментальные принципы остаются теми же: инструментировать, измерять, оповещать и итерировать.

## FAQ

### Сколько времени нужно для инструментирования существующего агента с Langfuse?

Для простого агента (один вызов LLM с одним инструментом) — примерно 1-2 часа. Для мультикомпонентного агента с несколькими инструментами и шагами рассуждения — полдня. Использование декоратора `@observe` и коллбэков LangChain/LlamaIndex значительно сокращает время, так как большинство вызовов захватываются автоматически.

### Что делать, если сам Langfuse выходит из строя?

SDK Langfuse разработан для отказоустойчивости: если сервер Langfuse недоступен, SDK ставит трассы в очередь и отправляет их при восстановлении сервера. Ваше приложение продолжает работать нормально. Для критических развёртываний используйте self-hosted режим с избыточной базой PostgreSQL.

### Как отличить настоящий баг от просто низкокачественного ответа?

Это важное различие. Баг — это некорректное поведение системы (техническая ошибка, бесконечный цикл, крах). Низкое качество — это технически корректный, но неудовлетворительный ответ. Langfuse помогает различать: баги проявляются через трассы с ошибками (статус error), а проблемы качества — через низкие оценки на технически успешных трассах.

### Генерируют ли оповещения Langfuse много ложных срабатываний?

Зависит от конфигурации порогов. Лучшая практика — начинать с широких порогов и постепенно уточнять, наблюдая ложные срабатывания. Использование скользящих окон (среднее по 50 трассам, а не по одной) значительно снижает ложные срабатывания. Предиктивные оповещения Langfuse v3 ещё больше уменьшают эту проблему.

### Можно ли использовать Langfuse с агентами вроде AutoGPT или OpenClaw?

Да, с определёнными адаптациями. Автономные агенты вроде [AutoGPT](/blog/autogpt-bilan-2026) или [OpenClaw](/blog/openclaw-test-exclusif) генерируют гораздо более длинные и сложные трассы, чем простой чатбот. Langfuse v3 с распределённой трассировкой особенно подходит для этих случаев. Рекомендуется настроить более агрессивный сэмплинг и ограничить глубину трасс для предотвращения перегрузки.
